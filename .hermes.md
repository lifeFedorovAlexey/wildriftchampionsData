# wildRiftChampions workspace

## Purpose

This workspace powers `wildriftallstats.ru`. Treat it as three related but independent Git repositories:

- `./` — frontend/orchestration repository; the Next.js application is in `ui/`.
- `wr-api/` — API and data/import pipeline; this is the source of truth for business data.
- `wr-chat/` — realtime WebSocket delivery/presence layer; it must not become the source of truth for messages or other business data.

Always check Git status in the repository being changed. A clean status in the root repository says nothing about `wr-api/` or `wr-chat/` because both are ignored by the root repository.

## Current priorities and contracts

- Read `MASTER_PLAN.md` for product priority and `ARCHITECTURE_TASKS.md` for architecture status before broad changes.
- Read `GUIDE_DOMAIN_CONTRACT.md` before changing guide slugs, lane normalization, imports, or guide API behavior.
- Keep public assets S3-first in production. Do not reintroduce donor URLs or legacy proxy/mirror URLs into client-facing payloads.
- Keep public user authentication disabled unless the task explicitly enables it with a dedicated `USER_SESSION_SECRET` shared by UI and API.
- Do not add audio/voice work until the text-chat MVP is complete unless explicitly requested.

## Runtime

- Use Node.js `>=22.12` for the UI. The local baseline is Node 22.
- Hermes terminal commands on this Windows machine run through bash. Prefer forward-slash paths such as `D:/wildRiftChampions/ui`.
- Never read, print, commit, or overwrite `.env` or `.env.local` secrets. Use the tracked example files to understand the environment contract.
- Do not run deploys, production imports, schema setup, data sync, cleanup workers, or commands that contact production unless the user explicitly asks.

## Commands and quality gates

### Root/frontend repository

Run from the workspace root:

```bash
npm run lint
npm test
npm run build
npm run verify
```

Equivalent UI commands run from `ui/`:

```bash
npm run lint
npm run test
npm run dup:check
npm run build
```

A local UI build may log connection failures for `127.0.0.1:3002` when the API is not running; this is acceptable only when the build still exits successfully. For end-to-end behavior, start the API and use the documented local environment values.

### API repository

Run from `wr-api/`:

```bash
npm run lint
npm run test
npm run dup:check
```

Tests that require a database or external source must use explicit fixtures or an explicitly configured local environment. Do not silently use production.

### Chat repository

Run from `wr-chat/`:

```bash
npm test
```

### Completion rule

Before declaring a change complete:

1. Run the narrowest relevant test first.
2. Run lint and the full test suite in every repository changed.
3. Run the UI production build for changes affecting UI, shared contracts, routing, environment handling, or dependencies.
4. Check Git diff and Git status separately in root, `wr-api/`, and `wr-chat/` when more than one repository was involved.
5. Report warnings and skipped checks explicitly.

## Change discipline

- Work from `main` unless the user requests another branch or a release branch.
- Do not commit, push, tag, deploy, or modify GitHub secrets without explicit user approval.
- Keep changes scoped to the repository responsible for the behavior.
- Preserve compatibility between canonical champion slugs and existing legacy aliases.
- Add or update tests for behavior changes. Documentation and project-context-only changes do not require new runtime tests, but all existing checks must stay green.
- Prefer relative Markdown links so documentation works on GitHub and on other machines.

## Releases

- Package versions are mirrored in each repository's `package.json` and `package-lock.json` where present.
- Release branches use `release/x.y.z`; stable tags use `vx.y.z`.
- Update `CHANGELOG.md` when preparing a release.
- Follow `RELEASE.md` for the frontend release process and the repository-specific README for API/chat deployment expectations.
